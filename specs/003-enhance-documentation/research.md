# 研究与架构决策

## 决策 1: 文档框架选择
- **决策**: 继续使用 VitePress。
- **理由**: 
    1. **技术栈一致性**: 项目本身使用 Vite 构建，团队熟悉 Vue 生态。
    2. **性能**: VitePress 生成静态站点，加载速度极快，且支持 Markdown 扩展，适合技术文档。
    3. **多版本支持**: 虽然 VitePress 原生不直接内置“多版本选择器”组件（像 Docusaurus 那样开箱即用），但通过目录结构（如 `/v1/`, `/v2/`）和自定义主题配置可以轻松实现，且社区有成熟方案。
- **替代方案**: 
    - *Docusaurus*: React 生态，虽然功能强大且原生支持多版本，但引入了新的技术栈依赖。
    - *GitBook*: 商业化产品，定制性不如静态站点生成器。

## 决策 2: 间隔字符串解析策略
- **决策**: 在 `Scheduler` 内部实现一个轻量级的解析函数，支持 `s` (秒), `m` (分), `h` (时), `d` (天)。
- **理由**: 
    1. **零依赖**: 避免引入 `ms` 或 `human-interval` 等额外库，保持包体积 < 10KB 的目标。
    2. **兼容性**: 仅需将解析后的毫秒数传递给现有的 `TimerStrategy`，无需修改底层计时逻辑。
- **替代方案**: 
    - *引入第三方库*: 增加包体积，且功能过剩（只需解析简单的时间单位）。

## 决策 3: 多版本文档实现
- **决策**: 采用基于目录的路径版本控制策略。
    - 根目录 `docs/` 对应最新版本（Next/Dev）。
    - 历史版本归档在 `docs/vX.Y.Z/` 目录下。
    - 通过 VitePress 的 `nav` 导航栏添加版本下拉菜单进行切换。
- **理由**: 结构清晰，利于 SEO，且部署简单（纯静态文件）。
- **替代方案**: 
    - *基于 Git 分支构建*: CI/CD 复杂，用户无法在同一站点同时搜索不同版本内容。

## 决策 4: 示例项目集成
- **决策**: 在 `examples/` 目录下维护独立的子项目，但在根目录 `package.json` 中提供快捷指令。
- **理由**: 确保示例是完整的、可独立运行的项目，模拟真实用户环境。
